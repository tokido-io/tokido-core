# tokido-core 1.0.3 — Design Spec

**Date:** 2026-04-19
**Target version:** 1.0.3
**Triggered by:** Integration feedback from Tokido SaaS platform

---

## Problem Statement

Integration with the Tokido SaaS platform revealed two categories of issues:

1. **A production bug** caused by split ownership of the `confirmed` flag — the provider sets it in `store()`, the engine redundantly sets it again via `update()`. If an implementor's `update()` does a full metadata replace (natural in DynamoDB), all keys absent from the update payload are wiped — including `accountName`. This caused two production failures and required hotfix PRs.

2. **A documentation gap** — the SecretStore callback sequence (what gets called, in what order, with what arguments, for each MFA operation) is not documented anywhere. Implementors had to decompile bytecode to understand the contract.

Source JARs were confirmed present (maven-source-plugin configured since 1.0.0). The `EnrollmentContext` thinness and typed context are acknowledged as future work, not addressed here.

---

## Design

### 1. Fix confirmed ownership (engine owns, provider does not)

**Root cause:** `confirmed` is set in two places by two actors.

- `TotpFactorProvider.enroll()` stores `confirmed=false` in the initial `store()` metadata call.
- `MfaManager.enroll()` then calls `secretStore.update(userId, factorType, Map.of("confirmed", false))` — redundant and dangerous.

**Fix:** Provider removes `confirmed` from its `store()` metadata entirely. The engine's `update(confirmed=false)` call becomes the sole, intentional setter. The engine owns the full confirmation lifecycle:

| Event | Actor | Call |
|-------|-------|------|
| enroll (requiresConfirmation=true) | Engine | `update(CONFIRMED, false)` |
| confirmEnrollment success | Engine | `update(CONFIRMED, true)` |
| verify (requiresConfirmation=true) | Engine | reads `CONFIRMED` flag |

The provider stores only its own domain state: `lastCounter`, `createdAt`, `accountName`.

This change is safe: the update() merge semantics (keys absent from the map are preserved) already exist in the contract. The engine's `update(confirmed=false)` post-store is now documented as intentional, not a surprise.

### 2. Add SecretStore metadata key constants

Raw string literals (`"confirmed"`, `"lastCounter"`, `"accountName"`, etc.) are scattered across `MfaManager`, `TotpFactorProvider`, and `RecoveryCodeProvider`. Implementors have no way to know which keys to expect.

**Fix:** Add a `Metadata` nested class inside `SecretStore` with `public static final String` constants for every key the library reads or writes:

```java
public interface SecretStore {
    class Metadata {
        public static final String CONFIRMED     = "confirmed";
        public static final String LAST_COUNTER  = "lastCounter";
        public static final String LAST_USED_AT  = "lastUsedAt";
        public static final String CREATED_AT    = "createdAt";
        public static final String ACCOUNT_NAME  = "accountName";
        public static final String HASHED_CODES  = "hashedCodes";
    }
    // ...
}
```

All internal usages switch to these constants. Implementors can reference `SecretStore.Metadata.CONFIRMED` to build typed storage schemas.

### 3. Document the SecretStore callback sequence

Add a top-level Javadoc block on `SecretStore` describing the exact sequence of calls made for each MFA operation. The documentation must be precise enough that an implementor can implement the interface without reading the engine source.

**Sequence to document:**

**`MfaManager.enroll(userId, factorType, ctx)`**
- `exists(userId, factorType)` → must return false, else `AlreadyEnrolledException`
- `provider.enroll()` → internally calls `store(userId, factorType, secret, metadata)`
- If `requiresConfirmation()`: `update(userId, factorType, {confirmed: false})`

**`MfaManager.confirmEnrollment(userId, factorType, credential)`**
- `load(userId, factorType)` → reads current state
- `provider.verify()` → no store calls
- On success: `update(userId, factorType, {confirmed: true})`

**`MfaManager.verify(userId, factorType, credential)`**
- `load(userId, factorType)` → reads current state
- `provider.verify()` → internally calls `update(userId, factorType, {lastCounter, lastUsedAt})` on success (TOTP), or `update(userId, factorType, {hashedCodes, lastUsedAt})` on success (recovery)

**`MfaManager.unenroll(userId, factorType)`**
- `exists(userId, factorType)`
- `provider.unenroll()` → no store calls
- `delete(userId, factorType)`

The `update()` method Javadoc must prominently state: **"Implementations MUST merge — keys absent from the map MUST be preserved. A full metadata replace is incorrect and will corrupt state."**

### 4. Strengthen MfaManager Javadoc

Add lifecycle documentation to `MfaManager` class-level Javadoc, and per-method docs explaining which store callbacks fire. Cross-reference SecretStore.Metadata constants where relevant.

## What is NOT in scope

- Typed `TotpEnrollmentContext` — future work
- API signature changes — all public interfaces unchanged
- New factor types
- Testing infrastructure changes

---

## Affected files

| File | Change type |
|------|-------------|
| `tokido-core-api/…/spi/SecretStore.java` | Add `Metadata` constants, rewrite class Javadoc |
| `tokido-core-engine/…/engine/MfaManager.java` | Strengthen method Javadocs, document lifecycle |
| `tokido-core-totp/…/totp/TotpFactorProvider.java` | Remove `confirmed` from `store()` metadata; use `Metadata` constants |
